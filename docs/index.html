How to read these charts
  • Different algorithms, different knobs. SereneDB is IVF (+ optional scalar/product/RaBitQ quantization), Qdrant is HNSW. The search-effort knob swept per engine is nprobe for SereneDB and hnsw_ef for Qdrant — they aren't the same unit, so compare the recall-vs-QPS curve, not points at equal effort.
  • "Best per recall band." Each point/bar is the highest-QPS configuration (across every quantization, cell count, segment-merge policy, and effort knob measured) whose achieved recall@10 falls in that band — ties broken by lower p50 latency. Bands are 0.05-wide up to 0.95 recall, then 0.01-wide above it, since that's where the two engines separate.
  • QPS is closed-loop at 32 concurrent clients — each connection fires its next query as soon as the previous one returns, same protocol for both engines.
  • Build time and index size are shown per engine, not head-to-head. SereneDB and Qdrant build fundamentally different structures (IVF cells vs. an HNSW graph) from different knobs, so a single bar-for-bar comparison would imply an equivalence that isn't there. Each engine gets its own panel instead.
  • SereneDB panels are pinned at nlist_factor=2 (6,325 IVF cells for 10M rows). RaBitQ was only swept at nlist_factor=1 in this run, so it doesn't appear here.
  • The build-time panel's bar is a decomposition, not a comparison. Each quantizer's bar stacks CREATE INDEX time underneath the extra VACUUM (COMPACT_TABLE) merge time on top — both numbers come from the same build (the compact settle policy). The index-size panel separately shows compact vs. no-compact per quantizer — no-compact skips the background segment-merge after loading, so it's smaller on disk but typically slower to search.
  • SereneDB's index size includes the base-table columnstore copy of the vectors (SereneDB indexes a table it also stores; Qdrant's index is index-only) — keep that in mind when eyeballing the two engines' panels side by side.
